# ТЗ — Телеграм-бот «Статья → картинки»

> Статус: **черновик**. Форм-фактор и архитектура согласованы. Есть открытые вопросы к заказчику (раздел 8) — без ответов на них не начинать кодить парсер и доставку.
> Дата фиксации: 2026-05-19

---

## 1. Задача

Друг (заказчик) пишет статьи, внутри которых есть ТЗ на картинки. Сейчас картинки рисуются вручную. Нужно автоматизировать: **скинул статью → получил готовые картинки по своим референсам**.

Бот достаёт из статьи задания на иллюстрации, генерит картинки через Higgsfield по референсному стилю и отдаёт готовый пакет.

## 2. Пользователь

- Основной: заказчик (не разработчик, пользуется с телефона).
- Доступ — по whitelist. Чужие в боте недопустимы (см. раздел 7, пункт про деньги).

## 3. Решение

**Форм-фактор: Телеграм-бот.** Выбран осознанно вместо CLI (барьер терминала для не-технаря) и веба (это уже платный продукт, не текущая задача).

Принцип: **ядро переиспользуемое, бот — обёртка ввода/вывода**. Если позже понадобится веб — ядро не переписывается.

### UX-флоу

```
Заказчик → боту:  [пересылает файл статьи]
Бот:  Принял. Нашёл N заданий на картинки. Стиль: <пресет>. Генерю? /go
Заказчик: /go
Бот:  ⏳ Очередь N задач...
Бот:  (обновляет сообщение) прогресс M/N
Бот:  ✅ Готово. [архив] + опц. ссылка на Google Drive
```

## 4. Архитектура

| Слой | Ответственность | Меняется при смене форм-фактора? |
|---|---|---|
| Парсер статьи | Текст → JSON `[{id, описание, размер}]`. Если ТЗ не размечено — через GPT в строгий JSON | Нет (ядро) |
| Промпт-билдер | ТЗ + рефы → промпт + `aspect_ratio` | Нет (ядро) |
| Очередь + воркер | Higgsfield `subscribe()` + polling, лимит конкурентности, retry | Нет (ядро) |
| Доставка | Сбор результата в архив / Google Drive | Частично |
| Telegram-слой | Приём файла/ссылки, прогресс, отдача | Да (обёртка) |
| Whitelist | Допуск только по user_id | Да (обёртка) |

## 5. Технологический стек

- **Язык:** Python (офиц. `higgsfield-client`) либо Node (`@higgsfield/client`). По умолчанию Python.
- **Парсинг ТЗ:** GPT (структурированный вывод в JSON).
- **Генерация:** Higgsfield — модель **Flux Kontext** (`flux-pro/kontext/max/text-to-image`, держит консистентность по референсу) или **Soul** (фотореализм).
- **Постобработка:** Pillow (ресайз под нужные размеры, формат, имена файлов).
- **Хостинг:** VPS 24/7 (дешёвого достаточно).

## 6. Higgsfield — специфика интеграции

- Официальный API есть, SDK активно поддерживается (релиз ноябрь 2025).
- Авторизация: API-ключ формата `KEY_ID:KEY_SECRET`.
- **API асинхронный, job-based с polling**: отправил задачу → `job_id` → опрашиваешь статус → получил URL → скачал. Это диктует архитектуру (фоновый воркер, не висящий хендлер).
- Доступ к API — только на платном тарифе **Creator**.
- Документация скудная — заложить время на reverse-engineering авторизации.

### 6a. SDK сверен с фактическим пакетом (2026-05-19)

Веб-док врала (предлагала несуществующий `higgsfield_client.subscribe(...)`). Сверено живой интроспекцией установленного пакета:

- Пакет: **`higgsfield-client==0.1.0`** (API 0.x нестабилен — при апгрейде пересверить).
- Импорт: `import higgsfield_client` → `higgsfield_client.AsyncClient(base_url='https://platform.higgsfield.ai', timeout=90.0, api_key='KEY_ID:KEY_SECRET' | None)`.
- Креды: аргумент `api_key="KEY_ID:KEY_SECRET"`, либо env (`HF_KEY="key:secret"` или `HF_API_KEY`+`HF_API_SECRET`).
- `await client.subscribe(application, arguments) -> JSON` — submit+polling внутри, ждёт завершения. `arguments` (НЕ `input`).
- **Рефы:** `client.upload_image(PIL.Image, format='jpeg') -> url` — реальный путь передать рефы заказчика в Higgsfield.
- **НЕ подтверждено вживую (нет ключа), требует сверки на первом платном прогоне:**
  - точная форма ответа (реализован устойчивый `_extract_image_url` с явной диагностикой вместо тихого падения);
  - точный набор полей `arguments` под модель и форма передачи рефов в `arguments`.

## 7. Обязательные требования (правила, не обсуждаются)

1. **Результат — только файлом/архивом, НЕ «фото».** Телеграм пережимает фото в JPEG → ассеты портятся. Слать как документ.
2. **Whitelist по user_id обязателен.** Бот в Телеграме доступен любому, кто узнает @username. Без whitelist чужие жгут платный API-бюджет заказчика.
3. **Генерация — в фоновом воркере с прогрессом.** Higgsfield асинхронный, пачка = минуты. Хендлер не висит в ожидании.
4. **Имена файлов = id задания из статьи.** Чтобы заказчик сопоставил картинку с местом в статье.
5. **Постобработка под нужные размеры** (ресайз/кроп) — модель отдаёт свои соотношения сторон, не итоговые.
6. **Обработка брака.** Модель иногда выдаёт мусор: retry. **По умолчанию 1 вариант на слот** (экономия, см. п.7), N вариантов — только по явной кнопке.
7. **Минимизация трат заказчика — требование, НЕ опция.** Higgsfield платный, объём 10–30 регулярно. Следствия в архитектуре:
   - (а) 1 генерация на слот по умолчанию; перегенерация — только вручную по кнопке;
   - (б) **кэш**: одинаковый блок (хеш заголовка+пунктов+типа+рефов) не генерится повторно;
   - (в) **смета до запуска**: бот сообщает «N генераций ≈ $X» и ждёт `/go`, не жжёт API молча;
   - (г) хостинг — бесплатный/дешёвый tier, не новый платный VPS;
   - (д) экономный профиль качества Higgsfield по умолчанию.

## 8. Ответы заказчика по блокерам (зафиксировано 2026-05-19)

1. **Формат входа:** ✅ **И файл, и ссылка.** Поддержать оба: `.docx`/`.md`/`.txt` файлом и ссылку (Google Doc / Notion / опубликованная статья).
2. **Разметка ТЗ:** ✅ Решено по реальному примеру — см. раздел 8a.
3. **Референсы:** ✅ **Два типа картинок — «инфографика» и «сюжетная».** Бот должен определять тип задания и генерить по соответствующему набору рефов. Рефы заказчик загрузит заранее (две папки: `refs/infographic/`, `refs/story/`).
4. **Выдача результата:** ✅ **ZIP-архив в чат** (документом, не фото). Google Drive не нужен.
5. **Хостинг / аккаунт Higgsfield:** ✅ решение отложено, но задано **жёсткое требование: минимизировать денежные траты друга.** Higgsfield Creator платный и обязателен для генерации (главная статья расходов). Хостинг — дешёвый/бесплатный tier, НЕ новый платный VPS. Развёрнуто в разделе 7 (новое требование) и 10.
6. **Объём:** ✅ **средне, регулярно — ~10–30 картинок за прогон, потоком.** Нужна очередь + retry + смета. Усиливает требование экономии (10–30 × цена × регулярность).

## 8a. Разбор примера статьи (8Blocks, «Дефляция токенов»)

Пример: `/Users/george/Downloads/Перевод про сжигание _ 8Blocks.docx`

- **Маркер задания на картинку — строка, начинающаяся с `Рис.`** Дальше на той же строке опциональный заголовок (`Рис. Функции Блокчейна`, `Рис. ТОКЕНЫ`, либо пусто `Рис. `), следом буллет-список `•` с содержанием картинки. Блок ТЗ = строка `Рис.` + последующие буллеты до пустой строки/нового заголовка.
- В примере **3 блока, все — тип «инфографика»** (перечни данных): преимущества инфляции; функции блокчейна; список токенов BNB/HTR/ONE/LSK/SC/ATOM/FTM с описаниями.
- **Сюжетная** в этой статье через `Рис.` не размечена — это шапочная картинка (нейтральный мем-кот). Вывод: сюжетные могут идти вне маркера `Рис.` либо отсутствовать. Уточнить у заказчика, как помечать сюжетные (отдельный маркер?) — добавлено в блокеры ниже.
- **Определение типа:** заголовок может быть пустым, тип не всегда ясен из маркера → классифицировать по содержимому блока (структурированный перечень данных → инфографика; описание сцены/образа → сюжетная). Кандидат на GPT-классификатор.

## 8b. Новые блокеры, всплывшие из примера (уточнить до кода)

7. **Текст на инфографике:** ✅ **MVP — вариант 10.B (всё нейросетью).** Сначала генерим инфографику целиком, оцениваем качество текста на реальном результате (статья 8Blocks), потом решаем, нужен ли гибрид 10.A. Текст-слой (этап 6) откладывается до этой оценки.
8. **Маркер сюжетных:** ✅ **Один маркер `Рис.` для всего.** Тип (инфографика / сюжетная) бот определяет сам по содержимому блока → классификатор типа обязателен.

## 9. Вне scope (сознательно НЕ делаем сейчас)

- Веб-интерфейс / SaaS / мультипользовательский режим.
- Редактор картинок, ручная правка внутри бота.
- Интеграция с CMS / автопостинг в статью.
- Биллинг, тарифы, личные кабинеты.

(Всё это — потенциальный платный «продуктовый» трек, отдельно от текущей задачи.)

## 10. Риски (зафиксированы один раз)

- API Higgsfield только на платном Creator — без оплаченного аккаунта проект не стартует.
- Скудная документация Higgsfield → время на reverse-engineering.
- Kontext держит стиль хорошо, но 100% попадание в реф не гарантирует → нужен человек-приёмщик или N вариантов.
- Стоимость на пачку = цена/картинку × объём × частота — посчитать до старта.
- **Текст на инфографике (РЕШЕНИЕ ЗА ЗАКАЗЧИКОМ, блокер 8b.7).** Нейросеть точные подписи рисует нечитаемо. Две дороги:
  - **A. Гибрид:** нейросеть рисует только фон/визуал по рефам, текст (названия токенов, пункты) накладывается программно поверх (Pillow/SVG-шаблон). Текст всегда чёткий и правильный, но визуал = «картинка + слой текста», не цельная иллюстрация. Больше работы.
  - **B. Только нейросеть:** генерим инфографику целиком. Быстрее, цельный стиль, но **текст почти наверняка будет кривой/с ошибками** — для статьи о токенах с конкретными тикерами это брак.
  - Сюжетных картинок эта развилка не касается (там текста нет).
  - **Решение 2026-05-19: MVP = B.** Гибрид A не делаем, пока не увидим реальный результат B на статье 8Blocks. Если текст неприемлем — возвращаемся к A.

## 11. Этапы реализации (грубо, уточнятся после ответов раздела 8)

1. Ядро: парсер статьи (файл `.docx`/`.md`/`.txt` + ссылка) → извлечение `Рис.`-блоков → JSON слотов `[{id, заголовок, пункты, тип}]`.
2. Ядро: классификатор типа задания (инфографика / сюжетная) по содержимому блока.
3. Ядро: промпт-билдер — выбор набора рефов по типу (`refs/infographic/` vs `refs/story/`) → промпт + `aspect_ratio`.
4. Ядро: Higgsfield-воркер (subscribe + polling + retry + конкурентность).
5. Ядро: даунлоадер + постобработка (имена = id, размеры).
5a. **Экономия (обязательно, см. 7.7):** кэш блоков по хешу + смета стоимости перед прогоном (бот: «N генераций ≈ $X», ждёт `/go`).
6. **ОТЛОЖЕН (выбран 10.B):** текст-слой для инфографики. Вернуться, только если текст в B неприемлем.
7. Сборка ZIP + обёртка: Telegram-слой (приём файла/ссылки, прогресс, выдача архивом-документом).
8. Обёртка: whitelist по user_id.
9. Деплой на дешёвый/бесплатный хостинг, прогон на реальной статье (8Blocks).
